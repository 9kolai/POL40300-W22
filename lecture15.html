Steps in a typical computational analysis project

Generally, we can split any project into 4 overall steps:

  1. Develop research question.
  2. Collect data.
  3. Data preparation (pre-processing)
  4. Analysis

Connection between steps:

Research question

A good research question:

  • ...is theoretically insteresting.
  • ...clearly defines concepts of interest.
  • ...clarifies a relationship between concepts of interest.
  • ...is not too broad, but also not too narrow.
  • ...allows room for interpretaion, but not ambiguity. It should be clear what you are trying to study.

Research question examples

  • Why do the chickens cross the road?
    • Too broad: The question does not address which chicken or which road.
  • How many chickens crossed Richard Wagner Strasse in Munich, on February 1st, 2023?
    • Too narrow: This question could be answered in one sentence and does not leave room for analysis.
  • What are some of the environmental factors that occurred in Munich between January and February 2022 that would cause chickens to cross Richard Wagner Strasse?

Inspired by https://sites.duke.edu/urgws/files/2014/02/Research-Questions_WS-handout.pdf

Collect data

  • Availability.
  • Data quality (validity and reliability)
  • Computational methods:
    • Web scraping
    • API's
    • Sensors
    • Downloading documents directly from source
    • Re-use datasets collected by others
    • Online experiments
    • Online surveys (asking questions)
  • Salganik's distinction between:
    • Readymade data
    • Custommade data

Data preparation (pre-processing)

  • Cleaning data: Handling missing values and data types (characters, numbers, dates, boolean/logical)
  • Importing text into R
  • Tokenisation (words, sentences, bigrams, n-grams, compound tokens)
  • Normalisation: Stemming and lowercase
  • Removing irrelevant tokens: Symbols, punctuation, stopwords...
  • Count the tokens: Document-feature-matrix
  • Weighting of features

Analysis

  • Summary statistics: Max, min, mean, median, range, quantiles, sd, unique values, frequency tables.
  • Text document statistics: lengths of documents (words, sentences), top features.
  • Lexical diversity: Unique words in documents.
  • Dictionary analysis
  • Compare documents: Keyness analysis
  • (Machine Learning: Supervised and unsupervised)

Possibilities with computational text analysis

Grimmer & Stewart's (2013) four principles of automated text analysis:

  1. All quantitative models of language are wrong - but some are useful. They can be useful in several ways:
    • Assigning documents into predetermined categories (replacing human coding).
    • Help discover new and useful categories.
    • Measuring theoretically relevant quantities.
  2. Quantitative models for text amplify resources and augment humans.
  3. There is no globally best method for automated text analysis.
  4. Validate, validate, validate!
    • For supervised methods: Demonstrate that machine classifications replicates human coding.
    • For unsupervised methods: Demonstrate that measures produced corresponds to concepts claimed.
    • For dictionary analysis: Both methods can be relevant.

Validate dictionary

Three key issues we want to be aware of:

  • Validity: Does the categories actually make sense and represent well the concept we are interested in?
  • Recall: Does the dictionary capture all the content related to concept?
  • Precision: Does the dictionary only capture the content related to concept?

So recall and precision could be measured as:

  • Recall: Proportion of all mentions of concept that is actually captured by our dictionary.
  • Precision: Proportion of all the content captured by the dictionary which actually relates to concept of interest.

Validation of dictionary

Example from Maerz (2019): Simulating pluralism: language of democracy in hegemonic authoritarianism.

  • Qualitative check of codings in sample of 110 speeches.
    • Random sample of 5% of speeches from each case, excluding those already used to construct dictionary.
    • Keywords-in-context search.
  • Frequent negations and ambigiuous terms excluded.

Final dictionary available from link in paper.

Approach to build a good dictionary:

  1. Identify "extreme texts" with known positions.
  2. Use word frequencies from each group to identify words that occur often in one group, but rarely in the other.
  3. Check these keywords in context to measure precision (possibly just take a random sample)
  4. Consider to remove ambiguous words (and then repeat step 3 to see if precision improved)
  5. Read a random selection of full texts to see if there are obvious words that you missed (and then repeat step 3-4).
  6. (Consider need to stem keywords.)

Rvest: html_node() vs html_element()

html_node() and html_nodes() functions are deprecated, meaning we should not actually use them.

Instead they have been replaced with html_element() and html_elements() which work in exactly the same way.

No plans to remove html_node() and html_nodes() from package.

So feel free to use either function. They do the same.

Typical mistake

Often students refer to variable names in subset functions with double quotes: 


inaugDfmPostWWII <- dfm_subset(inaugDfm, "Year" >= 1945)

Never use double quotes when referring to a variable name in dfm_subset()!
(or the normal subset() function for that matter)

Be particular aware of this, since it will not return an error and might look like it works!

The correct subsetting should look like this: 


inaugDfmPostWWII <- dfm_subset(inaugDfm, Year >= 1945)
